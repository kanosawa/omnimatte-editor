# 01. プロジェクト概要

## 1.1 アプリの目的

ユーザーが読み込んだ動画ファイルに対して、キャンバス上でバウンディングボックスを指定するだけで、SAM2 によって対象物体のセグメンテーションマスクを動画全体にわたって取得し、原動画に重畳表示するデスクトップアプリケーション。

最終的には Omnimatte 系の手法へ発展させるためのフロントエンドとして機能することを想定しているが、本仕様の範囲は「mp4 入力 → BBox 指定 → SAM2 マスク取得 → 重畳表示」までとする。

## 1.2 機能要件

### コア機能（必須）

- F1: ローカルの mp4 ファイルを読み込み、キャンバスに表示する
- F2: 動画の再生／停止／コマ送り／コマ戻し／シーク操作
- F3: 動画停止中に、キャンバス上でマウス操作によるバウンディングボックス指定
- F4: SAM2 によるセグメンテーションを実行し、原動画にマスクを半透明＋着色合成した mp4 をバックエンドから取得
- F5: 取得した合成済み mp4 をフロントの `<video>` で再生して重畳結果を表示
- F6: 1物体のみ対応。新規BBox指定→SAM2実行で前の合成動画は破棄
- F7: `vendor/gen-omnimatte-public` を用いて、SAM2 で抽出した前景マスクに該当する物体を **base video から削除** した動画を生成する
- F8: トップバーの「SAM2 で検出」ボタンの **横** に「前景削除」ボタンを追加する
- F9: 「前景削除」ボタンは **SAM2 抽出が完了している場合のみ** 活性化する（マスクが手元にあり、合成 mp4 が表示されている状態）
- F10: 前景削除処理は **マスク重畳済みの合成 mp4 ではなく、base video**（前景削除前のオリジナル）に対して行う
- F11: 前景削除完了後、SAM2 で取得していたマスクは **無効化** する（重畳表示も解除し、`bbox` も `null` に戻す）
- F12: 前景削除後の動画に対して、**再度 BBox 指定 → SAM2 → 前景削除** を実行できる
- F13: 二回目以降の前景削除は、**直前の前景削除で生成された動画** を base video として実行する（カスケード）

### 横断的要件

- N1: バックエンドは原則 `127.0.0.1` バインドのシンプル構成。クラウド GPU サーバ運用時はクライアントから SSH ポート転送経由で接続する
- N2: バックエンドは AI モデルを起動時にバックグラウンドロード。ロード完了前にリクエストを受けても、完了を待ってから処理
- N3: SAM2 推論中も UI が固まらないこと（非同期処理）
- N4: mp4 はバックエンドにバイナリで送信する（パスを送らない）

## 1.3 非機能要件

- パフォーマンス: 動画再生時のフレーム描画は 30fps 以上を目標。ただし SAM2 推論時間は対象外
- プラットフォーム: Windows / macOS / Linux（electron 動作環境）
- 開発環境: Windows 11 を主ターゲットとし、Bash と PowerShell の両方で起動できること
- 配布: 本仕様の範囲では開発ビルドのみ。インストーラ生成は対象外

## 1.5 動作モード

バックエンドは `127.0.0.1:8000` で待ち受ける（既定）。ポートは `OMNIMATTE_PORT` 環境変数で変更可能。フロントエンドは `VITE_API_BASE` 環境変数（既定 `http://localhost:8000`）でバックエンドURLを参照する。

加えて、前景削除（Casper）は **同じ GPU マシン内の sidecar プロセス**（`127.0.0.1:8765` 既定。`OMNIMATTE_CASPER_PORT` で変更可）として起動し、本サーバから HTTP で呼ばれる。本サーバの lifespan が sidecar を自動 spawn するため、ユーザー操作は `python run.py` 1 コマンドで変わらない。フロントから sidecar は見えない。

- ローカル運用: 同一マシン上でバックエンドを起動 → そのまま `localhost:8000` に接続。sidecar も同マシンの `127.0.0.1:8765` で自動起動
- GPU サーバ運用: クラウドの GPU 機にバックエンドを置き、クライアント側で SSH ポート転送（`ssh -L 8000:127.0.0.1:8000 ...`）で `localhost:8000` を GPU サーバに張り替える。**sidecar は GPU 機の内部に閉じる**ので追加のポート転送は不要。フロントエンドの設定は変えない

認証はサーバ側のネットワーク制限（SSH のみ開放）で担保し、アプリ側に認証層は持たない。

## 1.6 主要な設計判断

実装時に迷わないよう、本プロジェクトで明示的に行った判断を記録する。

| 判断 | 内容 | 理由 |
|---|---|---|
| マスク返却方式 | サーバ側で原動画にマスクを半透明＋着色合成済みの mp4 を返す（マスク単体ではなく） | フロントは `<video>` 要素 1 本で再生するだけになり、原動画とマスク動画の同期問題が原理的に発生しない |
| mp4 送信方式 | パスではなくバイナリで送る | クラウドバックエンドでもパスは無効。送信方式を一本化 |
| Pixi の責務分離 | Pixi 操作は専用クラスに閉じ込め、React は薄いラッパ | キャンバス処理が React レンダリングサイクルに依存しないようにし、テスト性を高める |
| 状態管理の単位 | VideoElement と zustand を同一ファイルで同期 | 同期ロジックの分散を防ぎ、video 状態の単一の真実を維持 |
| 物体数 | 常に1物体 | 仕様簡素化。複数物体は将来拡張 |
| 前景削除の入力 | マスク重畳済み合成 mp4 ではなく、base video | マスクの色付けが推論ノイズになるのを避ける（F10） |
| 前景削除のカスケード | `/remove` 完了でセッションのベース動画自体を差し替える | 「2回目以降の前景削除は前回の結果に対して行う」要件 F13 を最小実装で満たす |
| マスクの保持位置 | フロントではなくサーバ（`MaskStore`） | マスクは大きく、JSON 化や HTTP 往復が無駄。`/remove` でも HTTP body にマスクを載せず空 POST で済む |
| Casper の起動方式 | 永続 sidecar プロセス（事前ロード）。本サーバとは別プロセスの FastAPI（`127.0.0.1:8765`）を lifespan で自動 spawn し、Casper をプリロード。本サーバから HTTP で呼ぶ | 事前ロードで `/remove` 応答は推論時間のみに短縮。プロセス分離で `videox_fun` の名前空間衝突、`gradio` 等の重依存混入、CUDA C 拡張 ABI 衝突といった依存衝突リスクを回避。クラウド GPU 機に sidecar だけを分離する運用 (`OMNIMATTE_SPAWN_CASPER=0` + `CASPER_SIDECAR_BASE`) にもそのまま乗る（`absl.flags` のモジュールトップ汚染は fork 版 `vendor/gen-omnimatte-public` で `if __name__ == "__main__"` ブロックに閉じ込めて解消済み） |
