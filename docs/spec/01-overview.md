# 01. プロジェクト概要

## 1.1 アプリの目的

ユーザーが読み込んだ動画ファイルに対して、キャンバス上でバウンディングボックスを指定するだけで、SAM2 によって対象物体のセグメンテーションマスクを動画全体にわたって取得し、原動画に重畳表示するデスクトップアプリケーション。

最終的には Omnimatte 系の手法へ発展させるためのフロントエンドとして機能することを想定しているが、本仕様の範囲は「mp4 入力 → BBox 指定 → SAM2 マスク取得 → 重畳表示」までとする。

## 1.2 機能要件

### コア機能（必須）

- F1: ローカルの mp4 ファイルを読み込み、キャンバスに表示する
- F2: 動画の再生／停止／コマ送り／コマ戻し／シーク操作
- F3: 動画停止中に、キャンバス上でマウス操作によるバウンディングボックス指定
- F4: SAM2 によるセグメンテーションを実行し、原動画にマスクを半透明＋着色合成した mp4 をバックエンドから取得
- F5: 取得した合成済み mp4 をフロントの `<video>` で再生して重畳結果を表示
- F6: 1物体のみ対応。新規BBox指定→SAM2実行で前の合成動画は破棄

### 横断的要件

- N1: バックエンドは原則 `127.0.0.1` バインドのシンプル構成。クラウド GPU サーバ運用時はクライアントから SSH ポート転送経由で接続する
- N2: バックエンドは AI モデルを起動時にバックグラウンドロード。ロード完了前にリクエストを受けても、完了を待ってから処理
- N3: SAM2 推論中も UI が固まらないこと（非同期処理）
- N4: mp4 はバックエンドにバイナリで送信する（パスを送らない）

## 1.3 非機能要件

- パフォーマンス: 動画再生時のフレーム描画は 30fps 以上を目標。ただし SAM2 推論時間は対象外
- プラットフォーム: Windows / macOS / Linux（electron 動作環境）
- 開発環境: Windows 11 を主ターゲットとし、Bash と PowerShell の両方で起動できること
- 配布: 本仕様の範囲では開発ビルドのみ。インストーラ生成は対象外

## 1.4 技術スタック

### フロントエンド

| 領域 | 採用技術 | 用途 |
|---|---|---|
| 言語 | TypeScript | 全フロントエンドコード |
| デスクトップ | Electron | デスクトップアプリ化 |
| ビルド | electron-vite | 開発サーバ・ビルド |
| UI フレームワーク | React | コンポーネント構築 |
| 状態管理 | zustand | グローバル状態 |
| 描画 | PixiJS | キャンバス描画（動画・マスク・BBox） |
| HTTP | fetch / axios（任意） | バックエンドAPI呼び出し |

### バックエンド

| 領域 | 採用技術 | 用途 |
|---|---|---|
| 言語 | Python 3.10+ | サーバ実装 |
| Web フレームワーク | FastAPI | HTTP サーバ |
| ASGI | uvicorn | プロセス起動 |
| 推論 | SAM2 (`sam2.build_sam2_video_predictor`) | セグメンテーション |
| 動画処理 | OpenCV / ffmpeg | mp4 のデコード／エンコード |
| 数値計算 | numpy, torch | 推論前後の処理 |

### 参照実装

`vendor/sam2/examples/segment_video_server.py` を出発点とし、API 形状とマスク返却方式を本仕様に合わせて改修する（詳細は [03-backend.md](03-backend.md) と [04-api.md](04-api.md)）。

## 1.5 動作モード

バックエンドは `127.0.0.1:8000` で待ち受ける（既定）。ポートは `OMNIMATTE_PORT` 環境変数で変更可能。フロントエンドは `VITE_API_BASE` 環境変数（既定 `http://localhost:8000`）でバックエンドURLを参照する。

- ローカル運用: 同一マシン上でバックエンドを起動 → そのまま `localhost:8000` に接続
- GPU サーバ運用: クラウドの GPU 機にバックエンドを置き、クライアント側で SSH ポート転送（`ssh -L 8000:127.0.0.1:8000 ...`）で `localhost:8000` を GPU サーバに張り替える。フロントエンドの設定は変えない

認証はサーバ側のネットワーク制限（SSH のみ開放）で担保し、アプリ側に認証層は持たない。

## 1.6 主要な設計判断

実装時に迷わないよう、本プロジェクトで明示的に行った判断を記録する。

| 判断 | 内容 | 理由 |
|---|---|---|
| マスク返却方式 | サーバ側で原動画にマスクを半透明＋着色合成済みの mp4 を返す（マスク単体ではなく） | フロントは `<video>` 要素 1 本で再生するだけになり、原動画とマスク動画の同期問題が原理的に発生しない |
| mp4 送信方式 | パスではなくバイナリで送る | クラウドバックエンドでもパスは無効。送信方式を一本化 |
| Pixi の責務分離 | Pixi 操作は専用クラスに閉じ込め、React は薄いラッパ | キャンバス処理が React レンダリングサイクルに依存しないようにし、テスト性を高める |
| 状態管理の単位 | VideoElement と zustand を同一ファイルで同期 | 同期ロジックの分散を防ぎ、video 状態の単一の真実を維持 |
| 物体数 | 常に1物体 | 仕様簡素化。複数物体は将来拡張 |
